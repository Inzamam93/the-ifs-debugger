IFS Aurena Frontend Debugging Tips That Will Save Your Sanity

Inzamam Ghazzali ‚Ä¢ 20 Mar 2025 ‚Ä¢ 12 min read

Aurena's client-side behaviour can feel like a black box. The UI does something unexpected, you get a vague error toast, and the only thing staring back at you is a minified JavaScript bundle. This article covers the practical debugging techniques I actually use ‚Äî not the sanitised official ones ‚Äî to trace frontend issues in IFS Cloud fast.

Your Best Friend: The Browser DevTools

If you're not already living in the browser's DevTools while debugging Aurena, start now. F12 (or Ctrl+Shift+I) is your debugger console, network inspector, and DOM explorer all in one. Most IFS developers underuse it ‚Äî here's what to actually look at.

The Console Tab

Aurena logs a surprising amount of useful information to the console if you know what to filter for. The noise-to-signal ratio is high, but filtering by Errors and Warnings first cuts through 80% of the noise. Look specifically for:

  • Uncaught Promise rejections ‚Äî these often surface projection call failures that don't make it to the UI toast
  • 404s on .js or .json resources ‚Äî a sign of a missing custom component or misconfigured deployment
  • CORS errors ‚Äî particularly relevant when debugging IFS Connect-backed pages
üí° Tip In Chrome DevTools, right-click the console filter area and enable "Verbose" level logging. Aurena emits additional projection lifecycle events at verbose level that are hidden by default. These are gold when tracing why a page isn't populating data.

The Network Tab

This is where most of your Aurena debugging will actually happen. Every interaction Aurena makes with the backend goes through the network tab as an XHR or Fetch request. Filter by Fetch/XHR and watch what fires when you trigger the broken action.

The key request to look for is the projection call ‚Äî it'll look something like this: 

// Typical Aurena projection request URL pattern
POST /main/ifsapp/projection/v1/YourProjectionName/actions/ActionName

// Or for entity reads:
GET  /main/ifsapp/projection/v1/YourProjectionName/EntityNameSet(key='VALUE')

Click on that request, go to the Response tab, and you'll see the raw JSON back from the server ‚Äî including any server-side error detail that the UI swallowed. This is usually more informative than whatever toast Aurena showed you.

‚ö† Watch Out Aurena often returns HTTP 200 even on logical errors ‚Äî the error detail is buried inside the JSON response body under an error or message key. Always check the response body, not just the status code.

Tracing Projection Calls

Understanding which projection is being called and with what parameters is half the battle. Aurena's OData-based projection API is verbose but readable once you know the pattern. Here's how to trace a full projection interaction.

Reading the Request Payload

For action calls (e.g. posting a transaction, running a command), click the network request and go to the Payload tab. You'll see the JSON body sent to the server: 

// Example action call payload ‚Äî Network ‚Üí Payload tab
{
  "selection": {
    "KeyRef": "ORDER_NO=ORD-12345^"
  },
  "parameters": {
    "PostingDate": "2025-03-20",
    "NoteText":    "Manually triggered via UI"
  }
}

This tells you exactly what data Aurena is sending to your projection action. If the action is failing, compare this against what your PL/SQL action procedure expects. Nine times out of ten, the mismatch is in parameter naming ‚Äî Aurena is case-sensitive and expects the parameter names to match the projection metadata exactly.

Enabling Aurena Debug Mode

IFS Cloud has a built-in debug mode for Aurena that surfaces additional runtime information. It's not prominently documented, but it's invaluable. Here's how to enable it.

URL Parameter Method

Append ?debug=true to your Aurena URL. Some IFS Cloud versions also support ?logLevel=debug. This enables:

  • Extended console logging for component lifecycle events
  • Projection metadata resolution logging
  • Client-side validation rule tracing
  • Additional error detail in toast messages
// Example: enabling debug mode on a specific page
https://your-ifs-instance.com/main/ifsapp/CustomerOrder?debug=true

// Or combined with a specific page context:
https://your-ifs-instance.com/main/ifsapp/CustomerOrder?debug=true&logLevel=verbose
üî¥ Gotcha Do not leave ?debug=true in bookmarks or share debug-mode URLs with end users. In some IFS Cloud configurations, debug mode bypasses certain client-side permission checks and can expose metadata that shouldn't be visible in production.

DOM Inspection for Binding Issues

When a field isn't populating, isn't editable, or is behaving unexpectedly, the Elements tab in DevTools is your next stop. Aurena renders its components with data- attributes that expose projection binding information directly in the DOM.

Finding Projection Binding in the DOM

Right-click the problematic field in Aurena and select Inspect. Look for attributes like: 

<ifs-field
  data-projection="CustomerOrderHandling"
  data-entity="CustomerOrderSet"
  data-attribute="WantedDeliveryDate"
  data-binding="twoWay"
  data-editable="false"
  data-visibility="visible"
>

See that data-editable="false"? That tells you the field is being made read-only by the projection metadata ‚Äî not by a client-side condition. This means you need to go to the server-side projection definition and check the attribute's editability conditions for the current entity state.

Querying the Metadata Endpoint Directly

This is the technique most developers don't know about, and it's one of the most powerful debugging tools in the Aurena arsenal. Every projection exposes a metadata endpoint that returns the full structural definition of the projection ‚Äî entities, attributes, actions, editability rules ‚Äî as JSON. 

// Query the projection metadata directly in your browser
GET /main/ifsapp/projection/v1/YourProjectionName/$metadata

// Example:
GET /main/ifsapp/projection/v1/CustomerOrderHandling/$metadata

Paste that URL in a new browser tab while logged into IFS. You'll get back an XML or JSON document (depending on your IFS version) describing every entity, attribute, action and their associated metadata ‚Äî including editability conditions, required flags, and state-based rules.

üí° Tip Install the JSON Formatter Chrome extension before doing this. The metadata response is large and unreadable without formatting. Once formatted, use Ctrl+F to search for the specific attribute or action you're debugging.

Debugging Client-Side Validation Rules

Aurena supports client-side validation rules defined in the projection that run before any server call is made. These are common culprits when a save or action is silently blocked ‚Äî the validation fails on the client, the server is never called, and the only indicator is a subtle field highlight or a vague toast.

Identifying Client-Side vs Server-Side Blocks

The quick way to tell them apart:

  • No network request fired when you click Save/OK ‚Üí client-side validation block
  • Network request fired but returns an error ‚Üí server-side validation block

If no request fires, go to the Network tab, trigger the action, and confirm nothing appears in the XHR/Fetch list. Then look at the console for validation-related log entries.
