IFS Aurena Frontend Debugging Tips That Will Save Your Sanity

Aurena's client-side behaviour can feel like a black box. The UI does something unexpected, you get a vague error toast, and the only thing staring back at you is a minified JavaScript bundle. This article covers the practical debugging techniques I actually use â€” not the sanitised official ones â€” to trace frontend issues in IFS Cloud fast.

Your Best Friend: The Browser DevTools

If you're not already living in the browser's DevTools while debugging Aurena, start now. F12 (or Ctrl+Shift+I) is your debugger console, network inspector, and DOM explorer all in one. Most IFS developers underuse it â€” here's what to actually look at.

The Console Tab

Aurena logs a surprising amount of useful information to the console if you know what to filter for. The noise-to-signal ratio is high, but filtering by Errors and Warnings first cuts through 80% of the noise. Look specifically for:

  • Uncaught Promise rejections â€” these often surface projection call failures that don't make it to the UI toast
  • 404s on .js or .json resources â€” a sign of a missing custom component or misconfigured deployment
  • CORS errors â€” particularly relevant when debugging IFS Connect-backed pages
ğŸ’¡ tip In Chrome DevTools, right-click the console filter area and enable "Verbose" level logging. Aurena emits additional projection lifecycle events at verbose level that are hidden by default. These are gold when tracing why a page isn't populating data.

The Network Tab

This is where most of your Aurena debugging will actually happen. Every interaction Aurena makes with the backend goes through the network tab as an XHR or Fetch request. Filter by Fetch/XHR and watch what fires when you trigger the broken action.

The key request to look for is the projection call â€” it'll look something like this: 

// Typical Aurena projection request URL pattern
POST /main/ifsapp/projection/v1/YourProjectionName/actions/ActionName

// Or for entity reads:
GET  /main/ifsapp/projection/v1/YourProjectionName/EntityNameSet(key='VALUE')
HTTP

Click on that request, go to the Response tab, and you'll see the raw JSON back from the server â€” including any server-side error detail that the UI swallowed. This is usually more informative than whatever toast Aurena showed you.

âš  watch out Aurena often returns HTTP 200 even on logical errors â€” the error detail is buried inside the JSON response body under an error or message key. Always check the response body, not just the status code.

Tracing Projection Calls

Understanding which projection is being called and with what parameters is half the battle. Aurena's OData-based projection API is verbose but readable once you know the pattern. Here's how to trace a full projection interaction.

Reading the Request Payload

For action calls (e.g. posting a transaction, running a command), click the network request and go to the Payload tab. You'll see the JSON body sent to the server: 

// Example action call payload â€” Network â†’ Payload tab
{
  "selection": {
    "KeyRef": "ORDER_NO=ORD-12345^"
  },
  "parameters": {
    "PostingDate": "2025-03-20",
    "NoteText":    "Manually triggered via UI"
  }
}
JSON

This tells you exactly what data Aurena is sending to your projection action. If the action is failing, compare this against what your PL/SQL action procedure expects. Nine times out of ten, the mismatch is in parameter naming â€” Aurena is case-sensitive and expects the parameter names to match the projection metadata exactly.

Checking the Response for Server Errors

When a projection call fails on the server side, the response typically looks like this: 

// Error response â€” look for this structure in the Response tab
{
  "error": {
    "code":    "IFS_ERROR",
    "message": "ORA-20110: Order ORD-12345 is not in a valid state",
    "innererror": {
      "errordetail": "Validate_Order_State__: State check failed for RELEASED",
      "stacktrace": "..."
    }
  }
}
JSON

The innererror.errordetail field is the one you want â€” it usually contains the PL/SQL procedure name that raised the exception, which gives you a direct line to the server-side code to investigate.

ğŸ’¡ tip Copy the full error JSON and paste it into a JSON formatter (jsonformatter.org works fine). The stacktrace field inside innererror will often give you the full PL/SQL call stack â€” much more useful than the one-liner the UI shows.

Enabling Aurena Debug Mode

IFS Cloud has a built-in debug mode for Aurena that surfaces additional runtime information. It's not prominently documented, but it's invaluable. Here's how to enable it.

URL Parameter Method

Append ?debug=true to your Aurena URL. Some IFS Cloud versions also support ?logLevel=debug. This enables:

  • Extended console logging for component lifecycle events
  • Projection metadata resolution logging
  • Client-side validation rule tracing
  • Additional error detail in toast messages
-- Example: enabling debug mode on a specific page
https://your-ifs-instance.com/main/ifsapp/CustomerOrder?debug=true

-- Or combined with a specific page context:
https://your-ifs-instance.com/main/ifsapp/CustomerOrder?debug=true&logLevel=verbose
URL

What to Look For in Debug Mode

With debug mode enabled, filter your console for [Projection] and [Metadata] prefixes. You'll see log entries like:

[Projection] Fetching metadata for: CustomerOrderHandling [Projection] Action resolved: ReleaseOrder â†’ POST /projection/v1/CustomerOrderHandling/actions/ReleaseOrder [Metadata] Attribute 'WantedDeliveryDate' marked as read-only in state RELEASED [Projection] Entity refresh triggered after action completion [Projection] Failed to refresh CustomerOrderSet â€” server returned 500

That last line â€” the refresh failure after a successful action â€” is a very common Aurena pattern that confuses developers. The action succeeded, but the subsequent entity refresh to update the UI failed. This is a separate issue and needs to be investigated independently.

ğŸ”´ gotcha Do not leave ?debug=true in bookmarks or share debug-mode URLs with end users. In some IFS Cloud configurations, debug mode bypasses certain client-side permission checks and can expose metadata that shouldn't be visible in production.

DOM Inspection for Binding Issues

When a field isn't populating, isn't editable, or is behaving unexpectedly, the Elements tab in DevTools is your next stop. Aurena renders its components with data- attributes that expose projection binding information directly in the DOM.

Finding Projection Binding in the DOM

Right-click the problematic field in Aurena and select Inspect. Look for attributes like: 

<!-- Example: Aurena input field DOM attributes -->
<ifs-field
  data-projection="CustomerOrderHandling"
  data-entity="CustomerOrderSet"
  data-attribute="WantedDeliveryDate"
  data-binding="twoWay"
  data-editable="false"
  data-visibility="visible"
>
HTML

See that data-editable="false"? That tells you the field is being made read-only by the projection metadata â€” not by a client-side condition. This means you need to go to the server-side projection definition and check the attribute's editability conditions for the current entity state.

Checking Conditional Editability

In IFS Buildplace, open the projection and find the attribute in question. Check for Dynamic Component Availability or State Conditions applied to the attribute. These are the server-side rules controlling whether Aurena renders the field as editable. If they look correct, the issue may be in the entity's current state value not matching what the condition expects.

Querying the Metadata Endpoint Directly

This is the technique most developers don't know about, and it's one of the most powerful debugging tools in the Aurena arsenal. Every projection exposes a metadata endpoint that returns the full structural definition of the projection â€” entities, attributes, actions, editability rules â€” as JSON. 

-- Query the projection metadata directly in your browser
GET /main/ifsapp/projection/v1/YourProjectionName/$metadata

-- Example:
GET /main/ifsapp/projection/v1/CustomerOrderHandling/$metadata
HTTP

Paste that URL in a new browser tab while logged into IFS. You'll get back an XML or JSON document (depending on your IFS version) describing every entity, attribute, action and their associated metadata â€” including editability conditions, required flags, and state-based rules.

ğŸ’¡ tip Install the JSON Formatter Chrome extension before doing this. The metadata response is large and unreadable without formatting. Once formatted, use Ctrl+F to search for the specific attribute or action you're debugging.

Debugging Client-Side Validation Rules

Aurena supports client-side validation rules defined in the projection that run before any server call is made. These are common culprits when a save or action is silently blocked â€” the validation fails on the client, the server is never called, and the only indicator is a subtle field highlight or a vague toast.

Identifying Client-Side vs Server-Side Blocks

The quick way to tell them apart:

  • No network request fired when you click Save/OK â†’ client-side validation block
  • Network request fired but returns an error â†’ server-side validation block

If no request fires, go to the Network tab, trigger the action, and confirm nothing appears in the XHR/Fetch list. Then look at the console for validation-related log entries.

Tracing the Specific Rule

With debug mode enabled (?debug=true), failed client-side validations log to the console with a [Validation] prefix:

[Validation] Rule failed: RequiredFieldCheck on CustomerOrderSet.WantedDeliveryDate [Validation] Action blocked: SaveOrder â€” 1 validation error(s) present Field: WantedDeliveryDate â€” value is null, required: true for state PLANNED

This tells you exactly which field, which rule, and which state is causing the block. Go to the projection in Buildplace and look at the Required condition on that attribute for the PLANNED state.
